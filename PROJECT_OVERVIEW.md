# FluenZ Project Overview

## Purpose

FluenZ is an English-learning platform with two parallel learning modes:

- `DEFAULT`: a shared curriculum managed by admins
- `PERSONALIZED`: an AI-generated path created from onboarding answers

The current work in progress focuses on the `versioned-default-catalog-admin` change, which adds versioned default content, admin management, and learner mode switching.

## Stack

- Backend: Spring Boot 3, Spring Security, Spring Data JPA, PostgreSQL
- Frontend: React, Vite, TypeScript, Zustand
- Infra: Docker Compose with `postgres`, `backend`, `frontend`

## Runtime

- Frontend: `http://localhost:5173`
- Backend: `http://localhost:8080`
- DB: PostgreSQL in Docker

Important commands commonly used in this repo:

- `docker compose up -d --build backend`
- `docker compose up -d --build frontend`
- `docker compose ps`
- `docker compose logs backend --tail=120`

## Core Domain Model

### Personalized Learning

Existing personalized tree:

- `LearningPath`
- `Topic`
- `Situation`
- `Chunk`
- `SubPhrase`

This path is tied to a user and generated by AI during onboarding.

### Default Catalog

Shared admin-managed tree:

- `DefaultCatalogVersion`
- `DefaultTopic`
- `DefaultSituation`
- `DefaultChunk`
- `DefaultSubPhrase`

This is versioned content for the default curriculum.

## Key Product Rules

- New users start in `DEFAULT` mode.
- Users can switch between `DEFAULT` and `PERSONALIZED`.
- If a user switches to `PERSONALIZED` without an active AI path, they should be routed to onboarding.
- Default catalog content is versioned.
- Users already assigned to an older published default version should remain on that version.
- Admin can manage the default catalog.

Note:

- The current implementation now allows CRUD on any version, not only `DRAFT`, because this was explicitly requested during implementation.
- OpenSpec may still mention draft-only editing in some places, so future agents should treat the implemented behavior as the current source of truth unless the product owner changes it again.

## Auth and Roles

User entity now includes:

- `role`: `USER | ADMIN`
- `preferredLearningMode`: `DEFAULT | PERSONALIZED`
- `activeDefaultCatalogVersion`

Admin route protection:

- Frontend route `/admin` requires `ADMIN`
- Backend `/api/admin/**` requires `ADMIN`

If promoting a user manually in DB:

```sql
UPDATE users
SET role = 'ADMIN'
WHERE email = 'user@example.com';
```

The user must log out and log back in after role changes.

## Current Admin Features

Implemented in the admin portal:

- list catalog versions
- create draft version
- preview version
- publish version
- CRUD `Topic`
- CRUD `Situation`
- CRUD `Chunk`
- CRUD `SubPhrase`
- upload image from local machine for:
  - situation thumbnails
  - sub-phrase images

Upload endpoint:

- `POST /api/admin/catalog/media/upload`

Uploaded files are served from:

- `/uploads/**`

Configured via:

- `app.upload.dir`
- `app.base-url`

## Current Learner Features

Implemented:

- dashboard mode switch between `DEFAULT` and `PERSONALIZED`
- default catalog rendered through the learner dashboard
- default progress separated from personalized progress
- default practice flow wired through backend
- explicit error message on mode-switch failure

## Database / Migration Notes

This repo currently relies on Hibernate `ddl-auto: update`.

Because the project has evolved against an existing DB, old databases may require manual fixes when new columns are introduced. During this work, the following manual DB backfill was required for existing users:

- add/fill `role`
- add/fill `preferred_learning_mode`
- ensure `active_default_catalog_version_id`

Future agents should be careful when modifying entity constraints on existing tables, especially `users`.
