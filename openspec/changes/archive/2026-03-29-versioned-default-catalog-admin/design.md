## Context

FluenZ currently models learning content as a single user-owned AI-generated path. That works for personalized onboarding, but it cannot support a shared admin-managed curriculum, local media uploads, versioned publishing, or a dedicated admin workspace. The new change introduces a second content source: a versioned default catalog controlled entirely by admins, while keeping personalized onboarding and AI path generation intact.

## Goals / Non-Goals

**Goals:**
- Add one shared `DEFAULT` curriculum with versioning, draft, preview, and publish flow
- Keep `PERSONALIZED` learning paths generated by onboarding as a separate mode
- Track progress separately for default and personalized learning
- Add admin roles and a separate admin portal
- Support full-tree CRUD for Topic -> Situation -> Chunk -> SubPhrase in the default catalog
- Support CSV import with partial success and row-level error reporting
- Support local backend-hosted uploads for situation thumbnails and sub-phrase images
- Preserve users on the default catalog version they started until they finish

**Non-Goals:**
- Replacing the current personalized onboarding flow
- Moving media storage to cloud infrastructure
- Multi-tenant catalogs or multiple shared default curricula
- Real-time collaborative editing for admin content
- Advanced workflow states beyond draft/preview/published

## Decisions

### 1. Split shared catalog content from personalized user paths
**Decision:** Introduce a dedicated default catalog model instead of reusing `LearningPath`.

Suggested structure:
```text
DefaultCatalogVersion -> DefaultTopic -> DefaultSituation -> DefaultChunk -> DefaultSubPhrase
LearningPath -> Topic -> Situation -> Chunk -> SubPhrase
```

**Rationale:** `LearningPath` is already tied to a `User` and a personalized onboarding flow. Reusing it for shared admin content would conflate ownership, publishing lifecycle, version retention, and progress semantics.

### 2. Keep one logical default curriculum, but publish it as immutable versions
**Decision:** Model the shared curriculum as a sequence of versions. Exactly one version can be published for new users at a time, but older published versions remain addressable for users already assigned to them.

**Rationale:** This satisfies the product rule that there is only one default path at the business level while allowing safe evolution without breaking in-progress learners.

### 3. Assign users to a default catalog version explicitly
**Decision:** Store a user-to-default-version assignment, either directly on `User` or through a dedicated enrollment table.

Suggested fields:
- `preferredLearningMode`: `DEFAULT | PERSONALIZED`
- `activeDefaultCatalogVersionId`

**Rationale:** New publishes must not automatically move users who are mid-course. Explicit assignment makes version retention deterministic.

### 4. Separate progress stores by content source
**Decision:** Keep personalized progress on the current personalized content tree, and add separate progress tables for default catalog entities.

Suggested approach:
- Keep `UserSubPhraseProgress` for personalized paths
- Add `UserDefaultSubPhraseProgress` for default catalog sub-phrases

**Rationale:** The two modes have different entity trees and life cycles. Sharing one progress table would complicate keys, ownership, and version retention.

### 5. Add roles and admin-only authorization
**Decision:** Extend users with a role enum at minimum: `USER`, `ADMIN`. Secure admin APIs and admin frontend routes separately.

**Rationale:** Admin-only content management, publishing, CSV import, and media upload require explicit authorization instead of relying on generic authenticated access.

### 6. Use a dedicated admin portal
**Decision:** Add a separate admin route space in the frontend, with its own layout and protected entry points.

Suggested route family:
- `/admin/login` if separate handling is needed later
- `/admin`
- `/admin/catalog`
- `/admin/catalog/versions/:id`
- `/admin/import`

**Rationale:** Admin workflows are structurally different from learner workflows and should not compete with the learning shell navigation.

### 7. Support preview before publish through draft versions
**Decision:** Admin edits happen against a draft version. Preview reads from that draft in the admin portal. Publishing promotes that version for future user assignments only.

**Rationale:** This gives admins confidence before release and matches the requirement that preview happens in the admin web app.

### 8. Use local backend file storage for default catalog media
**Decision:** Store uploaded default-catalog media files under a backend-managed local directory and expose them through static file serving.

Suggested scope:
- `DefaultSituation.thumbnailUrl`
- `DefaultSubPhrase.imageUrl`

**Rationale:** This meets the current infrastructure constraint and keeps implementation simple. The storage abstraction can later move to cloud if needed.

### 9. CSV import supports partial success
**Decision:** Import the full tree from CSV and process row by row. Valid rows are persisted into the target draft version; invalid rows are skipped and returned in an import report.

Suggested report fields:
- total rows
- imported rows
- skipped rows
- error list with row number, raw identifier values, and message

**Rationale:** This matches the expected admin workflow and avoids forcing a full retry for minor spreadsheet issues.

### 10. Personalized mode remains AI-owned, but now includes situation thumbnails
**Decision:** Keep onboarding and path generation behavior, and extend personalized image generation to fetch/store image URLs for both situations and sub-phrases.

**Rationale:** The current app already generates sub-phrase imagery. Extending to situation thumbnails improves the dashboard and keeps personalized mode visually richer than the current text-only cards.

### 11. Default mode is the first-time learner entry point
**Decision:** New users should land in `DEFAULT` mode by default. If they switch to `PERSONALIZED` and do not yet have an active personalized path, route them to onboarding.

**Rationale:** This matches the product requirement and reduces initial friction while preserving the premium personalized branch.

## Risks / Trade-offs

- **Model duplication between default and personalized trees**
  This increases entity count, but keeps ownership and version semantics clear.

- **Local file storage complicates deployment**
  The first implementation is simpler, but production durability and backup need follow-up planning.

- **Version retention increases read-path complexity**
  API resolution must carefully load content from the user's assigned default version instead of always loading the latest published version.

- **CSV import can create partial catalogs**
  Partial success is desired, but import UX must clearly show what failed so admins can fix gaps before publishing.

- **Admin UI scope is large**
  The admin portal should be delivered in phases: read/preview first, then CRUD, then import, then publishing polish.

## Open Implementation Notes

- Prefer introducing new DTO families for default catalog responses rather than overloading personalized DTOs with many nullable fields.
- Consider an internal service layer split:
  - `DefaultCatalogService`
  - `DefaultCatalogVersionService`
  - `DefaultCatalogImportService`
  - `DefaultCatalogMediaService`
  - `LearningModeService`
- For local media serving, define a storage root that survives app restarts in development environments where possible.
