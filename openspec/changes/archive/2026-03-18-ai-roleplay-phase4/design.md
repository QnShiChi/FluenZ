## Context

Phase 4 is the final practice phase before session summary. It transitions users from structured drills into free-form conversation via a 5-step roleplay flow.

## Goals / Non-Goals

**Goals:**
- 5-step sequential roleplay chat (system intro → AI asks → user answers → AI replies → user asks → AI evaluates)
- Dynamic LLM prompting with context from DB (roles, situation, target chunks, turn rules)
- Chat UI with mic-only input (Web Speech API)
- Seamless continuation after Phase 3 steps

**Non-Goals:**
- No score gate (≥70) in roleplay — it's conversational
- No persistent chat history in DB — session-only
- No streaming responses (full response only)

## Decisions

### 1. AI first message: reuse `contextQuestion`
**Choice**: Use existing `chunk.contextQuestion` as the AI's first message in roleplay.
**Why**: Already generated by LLM during path creation. Zero latency. No new DB field needed.

### 2. Real-time LLM for Steps 4 & 5
**Choice**: New endpoint `POST /api/practice/roleplay/chat` that accepts chat history + turnNumber, builds dynamic prompt, calls OpenRouter LLM.
**Why**: Multi-turn conversation requires fresh LLM responses. Cannot be pre-generated.

### 3. Dynamic System Prompt structure
**Choice**: Build prompt dynamically from DB data:
- **Core context**: AI role, user role, situation description, target chunks
- **Turn rules**: Different instructions for turn 1 (acknowledge, max 2 sentences, no questions) vs turn 2 (answer + evaluate in Vietnamese)
**Why**: Prevents hallucination, keeps responses focused and concise.

### 4. Chat UI inside PracticePage (not a separate page)
**Choice**: Render chat bubbles within existing PracticePage when step type is `ROLEPLAY_CHAT`.
**Why**: Seamless continuation — no navigation break. Reuses existing mic/TTS infrastructure.

### 5. TTS for system cues in Vietnamese
**Choice**: Use Web Speech API with Vietnamese voice for system cues ("Hãy trả lời câu hỏi sau nha!", "Đến lượt bạn hỏi mình đó").
**Why**: Browser-native, no additional API cost.
