## ADDED Requirements

### Requirement: Two-pass generation pipeline
The system SHALL implement a phased generation pipeline: blueprint generation first, detailed content generation second, usable text-path publication third, and thumbnail hydration asynchronously after text publication.

#### Scenario: Pass 1 blueprint generation
- **WHEN** user triggers path generation
- **THEN** the system makes a single LLM call that returns a persona summary and a roadmap blueprint for the personalized path

#### Scenario: Pass 2 batched detail generation
- **WHEN** Pass 1 blueprint is complete
- **THEN** the system generates detailed content in batches or equivalent units, with each call including the persona context from Pass 1
- **AND** the system validates the returned content before publishing it as usable text

#### Scenario: Publish usable text path before thumbnails finish
- **WHEN** text content has passed contract validation
- **THEN** the system may publish the personalized path before all thumbnails are hydrated
- **AND** thumbnail enrichment continues asynchronously afterward

### Requirement: Strict content contract
Each generated personalized situation SHALL contain exactly 5 chunks. Each chunk SHALL contain 1 base sentence (root sentence with blanks) and exactly 3 validated variable phrases (SubPhrases). No more, no fewer.

#### Scenario: Content contract verification
- **WHEN** a batch of situations/content units is generated by the LLM
- **THEN** each situation has exactly 5 chunks, each chunk has exactly 3 variableChunks, and any output not matching the contract is repaired or retried narrowly

### Requirement: Generation produces 20-25 topics
Each personalized learning path SHALL contain between 20 and 25 topics, organized in a logical difficulty progression from easy to advanced concepts relevant to the learner's profile.

#### Scenario: Path with sufficient depth
- **WHEN** a new personalized path is generated
- **THEN** the resulting path contains at least 20 and at most 25 topics

### Requirement: Real-time generation progress
The system SHALL provide a progress endpoint that reports generation status (current phase, total steps, current topic/situation name, text-ready state) so the frontend can display a progress bar.

#### Scenario: Progress polling during generation
- **WHEN** the frontend polls `GET /api/onboarding/generation-progress/{pathId}` during generation
- **THEN** the response includes phase-aware progress fields indicating whether blueprint generation, detail generation, text publication, or thumbnail hydration is in progress

#### Scenario: Progress display in UI
- **WHEN** generation is running  
- **THEN** the frontend displays a progress bar with phase-aware status such as "Generating content..." and "Hydrating thumbnails..."
- **AND** the UI can transition once the usable text path is ready even if thumbnail hydration continues

### Requirement: Batch retry on failure
The system SHALL retry failed generation units up to 3 times. Successfully generated batches or units SHALL be preserved even if subsequent units fail.

#### Scenario: Partial generation recovery
- **WHEN** batch 3 of 5 fails after 3 retries but batches 1, 2 were successful
- **THEN** the already valid generated content is preserved
- **AND** the system retries only the failed unit rather than restarting the whole path
